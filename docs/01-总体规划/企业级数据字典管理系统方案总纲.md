# 企业级数据字典管理系统方案总纲

## 1. 文档目的

本文档用于定义企业级数据字典管理系统的建设总纲，作为后续立项、产品需求拆解、技术架构设计、实施计划和治理制度设计的基础输入。

本系统不是单一的数据字典工具，而是面向企业级数据架构治理、统一元数据管理、数据血缘分析和数据资产责任运营的综合平台。平台建设将结合 DAMA-DMBOK2 的数据管理能力框架与 TOGAF 10 的企业架构治理方法，形成可落地、可运营、可持续演进的企业数据治理基础设施。

## 2. 建设背景

企业当前数据管理面临以下典型问题：

- IT 系统多、建设周期长、技术栈异构，系统之间的数据结构、接口关系和数据流向缺少统一视图。
- 现有数据平台主要承担报表和数据加工职能，尚未形成对源系统建设、模型设计和架构治理的反向约束。
- 系统清单、数据模型、接口清单、报表口径、指标定义、责任人等架构资产薄弱，依赖人工经验和分散文档。
- 数据变更缺少影响分析能力，字段、表、接口、ETL、报表、指标之间的依赖关系不透明。
- 数据标准、业务术语、模型设计和物理实现之间缺少可追溯映射，导致口径不一致、重复建设和治理成本上升。
- 架构评审与数据治理之间缺少系统化闭环，难以将治理要求前置到系统建设和变更流程中。

## 3. 建设目标

平台建设目标是打造企业级数据架构治理与元数据运营系统，支撑全企业范围的数据资产可见、模型可管、标准可控、血缘可查、责任可追溯和变更可评估。

核心目标包括：

- 建立企业统一数据资产目录，覆盖系统、数据库、表、字段、接口、ETL、报表、指标和文件等资产。
- 建立概念模型、逻辑模型和物理模型的一体化管理体系，支撑业务语义、架构设计和技术实现之间的映射。
- 建立数据标准、业务术语、指标口径和模型对象之间的关联关系，提升定义一致性。
- 建立数据血缘和影响分析能力，首期覆盖系统级、接口级、表级、报表级和关键字段级血缘。
- 建立架构治理闭环，将数据模型、数据标准、血缘影响、责任人和安全分级纳入系统建设与变更评审。
- 建立数据资产责任体系，明确数据 Owner、数据 Steward、系统 Owner、应用负责人和数据平台责任人。
- 建立自动采集与人工治理结合的元数据运营机制，保证技术现状与治理定义同步演进。

## 4. 平台定位

本平台定位为企业数据架构治理与元数据运营平台。

平台同时承担三类职能：

- 架构治理平台：支撑系统新建、系统改造、数据模型变更、接口变更、报表新增和指标新增的架构评审。
- 元数据管理平台：汇聚并管理企业技术元数据、业务元数据、管理元数据和操作元数据。
- 数据资产运营平台：支撑资产目录、责任认领、标准执行、血缘分析、质量检查和治理运营。

平台不替代源系统、数据库、数据仓库、ETL 工具或 BI 工具，而是通过采集、治理、建模、关联和流程能力，将分散资产纳入统一治理框架。

| 迁移规划 | 按业务域和系统群分批纳管，形成治理能力成熟路线 |

## 6. 建设原则

### 6.1 全企业统一框架，分域推广

平台采用全企业统一元模型、统一标准、统一流程和统一权限框架，但实施节奏采用分业务域、分系统群推广。避免一次性行政填报导致数据质量不可控。

### 6.2 新平台作为治理权威源

平台作为数据定义、业务术语、数据标准、模型关系、责任人、血缘确认结果和架构审批结果的治理权威源。

源系统仍作为运行结构的事实来源，数据库、ETL、接口、BI 等现状通过自动采集进入平台，形成运行态视图。

### 6.3 自动采集与人工治理结合

技术元数据优先自动采集，业务定义、责任人、标准映射、概念模型、逻辑模型和架构意图通过人工治理流程补齐。

### 6.4 架构治理优先，血缘和字典协同演进

首期优先建立架构治理闭环，把平台嵌入系统建设和变更流程。统一元数据和血缘分析作为治理闭环的基础能力同步建设。

### 6.5 以闭环验收，而非只以资产数量验收

平台建设不能只统计表、字段、系统数量。首期验收必须证明系统能支持架构评审、模型变更、标准引用、血缘影响分析和责任追溯。

## 7. 总体能力规划

### 7.1 数据资产目录

统一管理企业数据相关资产，包括：

- 业务域、数据域、主题域。
- 系统、应用、模块。
- 数据库、Schema、表、字段、索引、约束。
- 接口、API、消息 Topic、文件交换。
- ETL 作业、调度任务、数据同步任务。
- 报表、看板、数据集、指标。
- 数据标准、业务术语、参考数据。
- 概念实体、逻辑实体、物理对象。

### 7.2 数据字典管理

数据字典管理需要覆盖技术结构和业务语义：

- 字段中文名、英文名、数据类型、长度、精度、是否主键、是否必填。
- 业务定义、业务规则、取值范围、枚举说明、默认值。
- 数据分类分级、敏感属性、脱敏策略。
- 数据 Owner、数据 Steward、系统 Owner、维护人。
- 标准映射、术语映射、指标映射。
- 来源系统、落地系统、上下游关系。
- 版本、状态、发布时间、更新时间。

### 7.3 数据模型管理

首期覆盖概念模型、逻辑模型和物理模型：

- 概念模型：业务概念、核心实体、实体关系、业务域归属。
- 逻辑模型：逻辑实体、逻辑属性、主外键关系、业务规则、标准引用。
- 物理模型：数据库表、字段、索引、约束、分区、存储位置。
- 映射关系：业务概念到逻辑实体，逻辑实体到物理表字段，物理字段到报表和指标。
- 模型版本：支持草稿、评审中、已发布、废弃等状态。
- 模型评审：纳入架构评审和变更审批流程。

### 7.4 数据标准与业务术语

平台应建立统一的数据标准和业务术语管理能力：

- 标准分类：基础标准、编码标准、命名标准、指标标准、主数据标准、参考数据标准。
- 标准状态：草稿、评审中、已发布、修订中、废止。
- 标准引用：字段、逻辑属性、指标、报表和接口必须支持引用标准。
- 标准检查：支持命名合规、类型合规、长度合规、枚举合规和定义完整性检查。
- 术语管理：支持业务术语、同义词、缩写、英文名、定义、责任部门和适用范围。

### 7.5 数据血缘与影响分析

首期血缘深度定位为表级和关键字段级：

- 系统到系统的数据流关系。
- 接口、文件、消息和 API 的数据交换关系。
- 数据库表之间的加工、同步和依赖关系。
- ETL 作业、调度任务、数据集、报表之间的依赖关系。
- 关键字段到指标、报表和下游字段的追溯关系。

影响分析需要支持：

- 表结构变更影响分析。
- 字段含义变更影响分析。
- 接口字段变更影响分析。
- ETL 任务失败影响分析。
- 报表口径变更影响分析。
- 指标定义变更影响分析。

### 7.6 架构治理流程

平台应嵌入以下治理流程：

- 系统新建架构评审。
- 系统改造架构评审。
- 数据模型新增和变更审批。
- 接口新增和变更审批。
- 报表和指标新增审批。
- 数据标准发布审批。
- 血缘确认和修正流程。
- 数据资产责任认领流程。
- 架构例外申请和豁免审批。

### 7.7 可视化与呈现

平台需要提供多层次图形化呈现：

- 企业数据架构总览图。
- 业务域数据地图。
- 系统间数据流图。
- 系统与数据库分布图。
- 概念模型关系图。
- 逻辑模型关系图。
- 物理表关系图。
- 表级血缘图。
- 关键字段血缘图。
- 报表溯源图。
- 变更影响分析图。

## 8. 总体架构建议

### 8.1 逻辑架构

平台建议采用分层架构：

| 架构层 | 主要能力 |
| --- | --- |
| 门户与体验层 | 资产搜索、数据地图、血缘图、工作台、治理看板、评审待办 |
| 业务能力层 | 数据字典、模型管理、标准管理、血缘管理、影响分析、资产目录 |
| 治理流程层 | 审批流、评审流、认责流、发布流、例外处理、审计追踪 |
| 元数据服务层 | 统一元模型、关系管理、版本管理、标签管理、质量检查 |
| 采集集成层 | 数据库采集、ETL 采集、BI 采集、接口采集、API 对接、批量导入 |
| 存储计算层 | 关系数据库、图数据库、搜索引擎、对象存储、任务调度 |
| 安全权限层 | 统一认证、角色权限、数据权限、敏感信息控制、操作审计 |

### 8.2 技术选型方向

平台化自研核心能力，结合开源协议友好的组件：

- 元数据主存储：关系型数据库，保存资产、模型、流程、版本和配置。
- 关系与血缘存储：图数据库或具备图查询能力的存储，用于血缘、模型关系和依赖分析。
- 搜索能力：全文检索引擎，用于资产、字段、术语、指标和报表搜索。
- 采集调度：任务调度框架，用于周期性元数据采集和解析任务。
- 流程引擎：支持审批、评审、发布和认责流程。
- 可视化组件：支持大规模图谱展示、过滤、聚合、路径分析和影响范围呈现。
- 开放 API：为架构管理、数据平台、DevOps、CMDB、IAM、BI 和数据质量平台提供集成能力。

## 10. 组织与运营机制

### 10.1 组织模式

采用“架构 + 数据共治”模式：

- 企业架构团队负责架构原则、架构蓝图、架构评审、系统依赖和合规检查。
- 数据治理团队负责数据标准、元数据质量、资产目录、血缘运营和责任体系。
- 系统团队负责源系统模型、接口、数据库结构和采集接入。
- 数据平台团队负责 ETL、数据仓库、数据集、报表和指标元数据接入。
- 业务部门负责业务定义、数据 Owner、指标口径和数据使用解释。

### 10.2 关键角色

| 角色 | 职责 |
| --- | --- |
| 企业架构师 | 制定架构原则，组织架构评审，维护企业架构视图 |
| 数据架构师 | 负责数据域划分、模型设计、模型评审和数据流设计 |
| 数据 Owner | 对数据定义、使用范围、质量和授权承担业务责任 |
| 数据 Steward | 维护数据字典、标准映射、责任信息和元数据质量 |
| 系统 Owner | 对系统资产、接口、物理模型和变更信息负责 |
| DBA | 支持数据库结构采集、物理模型校验和技术元数据确认 |
| 数据平台负责人 | 负责 ETL、调度、数据集、报表和指标血缘接入 |
| 安全合规负责人 | 负责数据分类分级、敏感字段和访问审计要求 |

## 11. 分阶段实施路线

### 11.1 第一阶段：治理框架与平台底座

目标是建立统一框架和最小可运行闭环。

主要工作：

- 定义企业统一元模型和资产分类体系。
- 建立业务域、数据域、系统、表、字段、接口、报表、指标等核心对象。
- 建立数据标准、业务术语、责任人和架构评审基础流程。
- 接入首批数据库、数据平台、BI 和接口资产。
- 建立资产搜索、数据字典、模型管理和基础血缘展示能力。
- 选择 1 到 2 个业务域作为首批纳管对象。

阶段产出：

- 企业数据资产目录雏形。
- 数据模型管理流程。
- 架构评审流程。
- 元数据采集任务。
- 首批血缘视图。
- 运营指标看板。

### 11.2 第二阶段：分域纳管与治理闭环

目标是按业务域复制推广治理机制。

主要工作：

- 按业务域或系统群分批纳管核心系统。
- 完善概念模型、逻辑模型、物理模型映射。
- 将系统建设、系统改造和数据模型变更纳入平台评审。
- 推动数据 Owner 和 Steward 认责。
- 完善标准引用和元数据完整性检查。
- 建立变更影响分析能力。

阶段产出：

- 多业务域数据地图。
- 模型评审闭环。
- 责任认领闭环。
- 标准执行看板。
- 变更影响分析报告。

### 11.3 第三阶段：血缘深化与自动化增强

目标是提升血缘准确性和自动化运营能力。

主要工作：

- 扩展 ETL、SQL、调度、BI、接口和 API 的解析能力。
- 建立关键字段级血缘确认机制。
- 支持血缘路径分析、上下游影响分析和报表溯源。
- 建立元数据质量规则和异常告警。
- 打通 DevOps、CMDB、IAM、数据质量和数据安全平台。

阶段产出：

- 表级血缘全景。
- 关键字段级血缘。
- 报表溯源能力。
- 元数据质量监控。
- 自动化采集和变更提醒。

### 11.4 第四阶段：企业级运营优化

目标是形成可持续运营的数据架构治理机制。

主要工作：

- 建立治理成熟度评估指标。
- 建立数据资产价值评估和复用分析。
- 建立架构债务、重复建设和标准偏离分析。
- 支持跨域数据共享和主数据治理。
- 支持管理层驾驶舱和治理绩效考核。

阶段产出：

- 企业数据治理运营体系。
- 架构合规报告。
- 数据资产价值看板。
- 数据复用和治理绩效分析。

## 12. 验收标准

### 12.1 治理闭环验收

平台必须能完成以下端到端闭环：

- 系统建设架构评审。
- 数据模型新增和变更审批。
- 数据标准引用检查。
- 数据资产责任认领。
- 表和关键字段血缘影响分析。
- 架构合规检查和审批记录追溯。

### 12.2 覆盖范围验收

每个首批业务域至少应完成：

- 核心系统清单纳管。
- 核心数据库、表、字段纳管。
- 关键接口和数据交换关系纳管。
- 关键报表、指标和数据集纳管。
- 核心概念模型和逻辑模型纳管。
- 数据 Owner、Steward、系统 Owner 责任明确。

### 12.3 血缘能力验收

首期血缘至少满足：

- 任一关键报表可追溯到上游数据集、ETL 任务、表和关键字段。
- 任一核心表变更可识别影响到的接口、任务、报表、指标和负责人。
- 任一关键字段可查看业务定义、标准映射、上下游关系和责任人。

### 12.4 元数据质量验收

关键资产必须具备：

- 中文名、英文名、业务定义、数据类型、责任人、所属系统。
- 数据分类分级、标准映射、更新时间、来源方式。
- 采集状态、确认状态、发布状态。

## 13. 关键风险与应对

| 风险 | 表现 | 应对策略 |
| --- | --- | --- |
| 架构资产基础薄弱 | 系统、接口、模型、责任人不完整 | 先建立盘点模板和最小必填项，分域补齐 |
| 一次性铺开质量失控 | 填报数据多但不可用 | 统一框架，分域纳管，按治理闭环验收 |
| 自动采集无法覆盖全部系统 | 老旧系统、非标准工具难解析 | 自动采集优先，人工确认补齐，逐步扩展适配器 |
| 血缘准确性不足 | SQL 动态拼接、手工加工、跨工具链断点 | 先做表级和关键字段级，建立人工确认机制 |
| 治理流程流于形式 | 平台与系统建设流程脱节 | 将架构评审和模型变更作为项目准入或上线前置条件 |
| 责任体系难落地 | Owner 和 Steward 不明确 | 结合组织职责和系统权责，建立认责流程和运营看板 |
| 自研范围过大 | 周期长、交付慢 | 核心元模型和流程自研，采集、图谱、搜索等能力优先复用开源组件 |
