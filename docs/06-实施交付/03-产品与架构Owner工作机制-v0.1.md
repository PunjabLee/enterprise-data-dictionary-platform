# 产品与架构 Owner 工作机制 v0.1

## 1. 目标

Product/Architecture Owner 负责定义“做什么、为什么做、边界在哪里、是否符合目标架构”，不负责替代 Development Agent 编码，也不负责替代 PR Steward Agent 做机械化质量检查。

本机制用于明确 Owner 的 Git 工作分区、工作清单、决策输出物和与多 Agent 开发流程的协作关系。

## 2. 角色边界

| 角色 | 主要职责 | 不负责 |
| --- | --- | --- |
| Product Owner | 产品范围、优先级、业务流程、验收标准、版本范围 | 具体编码、测试脚本实现 |
| Architecture Owner | 技术架构、应用架构、数据架构、集成架构、ADR | 代替开发 Agent 完成功能代码 |
| PR Steward Agent | PR 质量门禁、合并顺序、冲突协调、检查结果汇总 | 重新定义产品范围或架构方向 |
| Development Agent | 在明确范围内实现代码、文档、测试、脚本 | 自行扩大范围或变更架构决策 |

首期项目中，Product Owner 与 Architecture Owner 可以由同一人或同一主控 Agent 承担，但必须在文档中区分“产品决策”和“架构决策”。

## 3. Git 工作分区

Owner 使用独立 worktree 和独立分支，避免与开发 Agent 或 PR Steward Agent 混用工作区。

推荐 worktree：

```powershell
git switch develop
git pull --ff-only
git worktree add ..\codex-owner -b owner/architecture-board develop
```

推荐分支：

| 分支 | 用途 | 合入目标 |
| --- | --- | --- |
| `owner/architecture-board` | Owner 工作板、决策台账、跨模块收口 | `develop` |
| `owner/product-scope` | 产品范围、优先级、验收标准调整 | `develop` |
| `owner/adr-update` | ADR、技术决策、架构约束调整 | `develop` |
| `owner/release-scope` | 发布范围、里程碑、上线门禁调整 | `develop` |

约束：

- Owner 分支只做范围、决策、架构和治理文档变更。
- Owner 分支不直接修改 `platform/**` 业务代码。
- Owner 分支如需修改工程配置，必须说明原因，并由 PR Steward Agent 做额外检查。
- Owner 分支也必须通过 PR 合入 `develop`，不得直接推送或直接合并。

## 4. Owner 写入范围

Owner 默认可写范围：

| 目录 | 说明 |
| --- | --- |
| `docs/00-文档治理` | 分支规范、编码规约、冻结清单、文档清单 |
| `docs/01-总体规划` | 方案总纲、路线原则 |
| `docs/02-产品需求` | PRD、功能清单、页面交互、验收标准 |
| `docs/03-架构设计` | 技术架构、应用架构、部署架构、数据库物理模型 |
| `docs/04-数据治理与元模型` | 元模型、血缘、标准、术语、安全、流程 |
| `docs/05-接口与集成` | API 契约、DTO、事件、导入导出规则 |
| `docs/06-实施交付` | 实施节奏、分工机制、风险和里程碑 |
| `docs/08-测试验收` | 验收标准、测试范围、测试用例清单 |
| `docs/09-会议纪要与决策` | ADR、会议决策、未决问题 |

Owner 非默认写入范围：

| 目录 | 处理方式 |
| --- | --- |
| `platform/**` | 原则上不改；确需修改时应拆给 Development Agent |
| `.github/**` | 仅限 PR 模板、CI 门禁策略、仓库治理规则 |
| `tools/**` | 仅限仓库治理脚本，必须说明安全影响 |

## 5. Owner 工作清单

### 5.1 任务启动前

Owner 必须完成：

1. 确认任务目标和不做范围。
2. 确认业务价值、优先级和版本归属。
3. 确认涉及的限界上下文和模块边界。
4. 确认是否影响数据库、API、权限、流程、部署、安全。
5. 将任务拆成可由单个 Agent 独立完成的 Work Package。
6. 为每个 Work Package 指定写入范围、前置依赖和验收口径。

输出物：

- Work Package 清单。
- 验收标准。
- 影响分析。
- 需要补充的 ADR 或产品决策。

### 5.2 开发进行中

Owner 必须跟踪：

1. 是否出现范围蔓延。
2. 是否出现跨模块边界不清。
3. 是否需要新增 ADR。
4. 是否需要调整优先级或拆分 PR。
5. 是否存在业务规则、权限、安全或合规疑问。

Owner 不应直接接管开发分支改代码，应通过 PR 评论、ADR 或追加 Owner PR 修正方向。

### 5.3 PR 评审前

Owner 检查：

| 检查项 | 判断标准 |
| --- | --- |
| 需求一致性 | PR 是否实现了被批准的需求 |
| 范围控制 | 是否夹带未批准功能 |
| 领域边界 | 是否破坏限界上下文 |
| 架构一致性 | 是否符合 DDD、模块化单体、DAO 分层、事件策略 |
| 数据一致性 | 是否符合元模型、物理模型、状态机、版本规则 |
| 权限安全 | 是否符合 RBAC、数据范围、敏感字段控制 |
| 验收标准 | 是否能按测试用例或验收清单验证 |

Owner 结论：

| 结论 | 含义 |
| --- | --- |
| `Approved by Owner` | 产品和架构方向可接受，交由 PR Steward 做质量门禁 |
| `Changes Requested by Owner` | 范围、模型、流程或架构需调整 |
| `Blocked by Decision` | 存在未决问题，需先形成 ADR 或会议决策 |

### 5.4 合并后

Owner 必须确认：

1. 文档和实现是否仍一致。
2. 后续 Work Package 是否需要调整。
3. 未决问题是否已关闭或转入问题台账。
4. 是否需要更新发布范围、演示脚本或验收清单。

## 6. Owner 与 PR Steward 的分工

| 场景 | Owner | PR Steward Agent |
| --- | --- | --- |
| 功能是否应该做 | 决策 | 不决策 |
| 架构方向是否正确 | 决策 | 检查是否符合已决策规则 |
| PR 是否可合并 | 给出业务/架构意见 | 给出质量/集成意见 |
| 冲突如何解决 | 处理领域和架构取舍 | 处理 Git 和集成顺序 |
| 测试是否充分 | 确认验收口径 | 检查测试是否执行 |
| 风险是否接受 | 决策 | 暴露风险和证据 |

原则：

- Owner 说清“正确方向”。
- PR Steward 证明“变更质量可控”。
- Development Agent 负责“按边界实现”。

## 7. 决策输出物

Owner 决策必须沉淀到以下文件之一：

| 决策类型 | 输出位置 |
| --- | --- |
| 技术栈、架构边界、中间件、分层规则 | `docs/09-会议纪要与决策/01-技术决策ADR-v0.1.md` |
| 产品范围、版本范围、功能优先级 | `docs/02-产品需求` 或 `docs/06-实施交付` |
| 数据模型、元模型、状态机、血缘规则 | `docs/04-数据治理与元模型` |
| API、DTO、事件、导入导出契约 | `docs/05-接口与集成` |
| 编码门禁、分支规则、协作规则 | `docs/00-文档治理` |

禁止只在聊天记录中形成关键决策。关键决策必须进入文档或 ADR。

## 8. Owner 看板建议

Owner 可以维护轻量看板，字段如下：

| 字段 | 说明 |
| --- | --- |
| ID | `OWN-001`、`OWN-002` |
| 类型 | 产品、架构、数据、安全、流程、交付 |
| 决策事项 | 需要确认的问题 |
| 当前结论 | 已确认、待确认、暂缓、废弃 |
| 影响范围 | 文档、后端、前端、数据库、部署 |
| 关联 PR | GitHub PR 编号或分支 |
| Owner | 责任人 |
| 截止时间 | 需要完成的日期 |

首期可以先不单独建看板文件，使用 ADR、冻结检查清单和 PR 评论承载；当并行 PR 超过 5 个时，应新增 Owner 看板文档。

## 9. 禁止事项

- 禁止 Owner 在未更新 ADR 的情况下改变关键技术路线。
- 禁止 Owner 直接要求 Development Agent 绕过既定分层规约。
- 禁止 Owner 将多个不相关需求打包给同一 Agent。
- 禁止 Owner 直接在 `develop` 或 `main` 修改文件。
- 禁止 Owner 在 PR 合并后才补验收标准。
- 禁止 Owner 以口头结论替代文档化决策。
