# 编码规约与架构约束 v0.1

## 1. 目标

本文档定义平台工程编码前必须遵守的架构约束和工程规约。所有后端、前端、数据库、接口、集成、测试和部署实现都应以本文档为红线。

## 2. DDD 分层规约

后端统一采用 DDD 分层：

```text
interfaces -> application -> domain
infrastructure -> domain/application ports
```

标准包结构：

```text
<bounded-context>/
  interfaces/
    controller/
    dto/
  application/
    service/
    command/
    query/
  domain/
    model/
    service/
    repository/
    event/
  infrastructure/
    repository/
    dao/
    mapper/
    po/
    converter/
```

硬性约束：

- `domain` 不允许依赖 Spring、MyBatis、Redis、MQ、HTTP 客户端等基础设施。
- `application` 不允许直接依赖 MyBatis Mapper。
- `interfaces` 不允许直接依赖 Mapper、DAO、RepositoryImpl。
- `infrastructure` 负责实现仓储、外部系统、消息、缓存、文件和数据库访问。
- 事务边界放在 `application` 层。
- 领域对象不处理事务、权限、HTTP、SQL、缓存和消息发送细节。

## 3. Repository / DAO / Mapper 规约

禁止 Service 直接访问 MyBatis Mapper。

推荐调用链：

```text
Controller
  -> ApplicationService
    -> DomainService / Aggregate
    -> Repository interface
      -> RepositoryImpl
        -> DAO
          -> MyBatis Mapper
```

职责划分：

| 对象 | 职责 |
| --- | --- |
| Repository Interface | 定义领域需要的持久化能力，面向聚合和领域对象 |
| RepositoryImpl | 实现 Repository，编排 DAO、Converter、缓存和数据权限 |
| DAO | 封装具体持久化查询和更新，面向 PO 和 Mapper |
| MyBatis Mapper | 只承载 SQL 映射，不承载业务逻辑 |
| PO | 数据库持久化对象 |
| DTO | API 入参出参对象 |
| Command | 写操作用例输入 |
| Query | 查询用例输入 |
| Assembler/Converter | 负责 DTO、Command、Domain、PO 之间转换 |

复杂列表、搜索、看板类查询允许走：

```text
Controller -> QueryService -> QueryRepository -> DAO -> Mapper
```

但仍不允许 Controller 或 Application Service 直接调用 Mapper。

## 4. 领域上下文规约

首期建议的限界上下文：

| 上下文 | 职责 |
| --- | --- |
| metadata | 元模型、资产、属性、关系、版本 |
| catalog | 资产目录、检索、标签、收藏 |
| dictionary | 表字典、字段字典、业务定义 |
| modeling | 概念模型、逻辑模型、物理模型、模型映射 |
| standard | 数据标准、业务术语、引用和整改 |
| lineage | 血缘关系、血缘确认、影响分析 |
| governance | 架构评审、模型变更、标准发布、责任认领 |
| collector | 数据源、采集任务、导入批次、差异识别 |
| security | 用户、角色、权限、数据范围、审计 |
| integration | 开放 API、消息、外部系统适配 |

模块之间禁止直接访问对方 Mapper、DAO 和数据库表。跨模块交互优先通过应用服务、领域事件或明确的端口接口完成。

## 5. API 规约

统一使用 REST + JSON。

统一响应结构：

```json
{
  "code": "0",
  "message": "success",
  "data": {},
  "traceId": "string"
}
```

分页结构：

```json
{
  "items": [],
  "page": 1,
  "pageSize": 20,
  "total": 0
}
```

API 约束：

- 写操作必须有权限校验和审计。
- 批量导入、采集、影响分析、批量导出必须异步。
- API 不直接暴露 PO。
- API 不返回底层异常堆栈。
- 状态冲突返回明确业务错误码。
- 查询接口必须定义分页、排序和筛选字段。

## 6. 异常与错误码规约

错误类型：

| 类型 | 示例 |
| --- | --- |
| 参数错误 | 必填缺失、枚举非法、格式错误 |
| 权限错误 | 未登录、无菜单权限、无数据范围权限 |
| 状态冲突 | 已发布对象直接编辑、审批中对象重复提交 |
| 资源不存在 | 资产、标准、任务、审批单不存在 |
| 导入错误 | 模板版本错误、字段缺失、重复编码 |
| 系统错误 | 数据库异常、外部系统不可用、未知异常 |

约束：

- 不允许把 SQL 异常、栈信息、数据库字段名直接暴露给前端。
- 导入校验错误必须能定位到行号、字段和错误原因。
- 异常日志必须包含 `traceId`。

## 7. 事务规约

- 事务边界放在 Application Service。
- 一个用例一个清晰事务边界。
- 跨聚合修改必须通过应用服务编排。
- 不允许在领域对象中开启事务。
- 不允许在长耗时任务中持有数据库事务。
- 消息发送与本地事务需考虑一致性，后续可引入 outbox 模式。

## 8. 权限与审计规约

权限模型：

```text
RBAC + 数据范围权限 + 敏感字段控制
```

必须审计的操作：

- 登录、登出。
- 新增、修改、删除、发布核心资产。
- 审批、驳回、撤回、转办。
- 导入、导出。
- 权限变更。
- 敏感字段查看。
- 采集连接配置变更。

约束：

- 权限判断集中实现，不允许散落在 Controller 或 SQL 片段中。
- 敏感字段默认受控展示。
- 导出必须校验权限并记录审计。
- 审计日志不得被普通用户删除。

## 9. 数据库规约

核心业务表必须具备：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `tenant_id` | 租户标识，首期默认 `default` |
| `created_at` | 创建时间 |
| `created_by` | 创建人 |
| `updated_at` | 更新时间 |
| `updated_by` | 更新人 |
| `deleted` | 软删除标识 |
| `version` | 乐观锁或业务版本 |

数据库约束：

- 核心编码必须有唯一约束。
- 高频查询字段必须有索引。
- 禁止生产环境手工改表。
- 所有 DDL 必须通过 Flyway 管理。
- 已执行的 Flyway 脚本禁止修改，只能新增。
- 核心业务数据采用软删除。

## 10. 缓存规约

Redis 用途：

- 热点字典。
- 用户权限快照。
- 短期任务状态。
- 分布式锁。
- 限流或幂等标记。

约束：

- Redis 不作为事实数据源。
- 缓存必须有 TTL 或明确失效机制。
- 缓存 key 必须统一命名。
- 权限和敏感数据缓存必须考虑失效和安全边界。

## 11. 领域事件与消息规约

定义统一端口：

```text
DomainEventPublisher
```

首期可使用本地事件；后续适配 RocketMQ 或 Kafka。

适合事件化的场景：

- 资产已发布。
- 模型变更已通过。
- 标准已发布。
- 采集任务完成。
- 导入批次完成。
- 血缘已确认。
- 影响分析完成。

约束：

- 业务代码不直接依赖 RocketMQ/Kafka 客户端。
- 消费者必须幂等。
- 消息失败必须可重试或进入失败表。
- 不在消息消费中持有不可控长事务。

## 12. 规则引擎规约

适合规则配置的业务：

- 元数据完整性检查。
- 标准映射检查。
- 架构评审检查项。
- 命名规范检查。
- 敏感字段识别。
- 数据分类分级建议。
- 影响等级判定。
- 审批路由建议。
- 治理质量评分。

不适合规则引擎的业务：

- 核心事务一致性。
- 资产 CRUD。
- 权限核心判断。
- 审批状态机。
- 血缘图遍历算法。

约束：

- 规则执行必须记录版本、输入、输出和命中结果。
- 规则建议不能绕过人工审批。
- 规则异常不得阻断核心查询能力。

## 13. 前端工程规约

前端按领域模块组织：

```text
src/modules/catalog
src/modules/dictionary
src/modules/modeling
src/modules/lineage
src/modules/governance
src/modules/security
```

每个模块内部：

```text
api/
model/
services/
views/
components/
```

约束：

- 禁止全局巨型 `api.ts`。
- Pinia 只保存跨页面状态。
- 表单临时状态留在组件内。
- API 类型必须与后端 DTO 对齐。
- 前端校验用于体验，后端校验才是权威。
- 大型页面必须拆分为容器组件、表单组件、表格组件和领域组件。

## 14. 测试与架构门禁

必须覆盖：

- 单元测试。
- Application Service 测试。
- API 集成测试。
- Flyway 迁移测试。
- 导入校验测试。
- 权限测试。
- 架构依赖测试。

建议引入 ArchUnit 规则：

```text
..domain.. must not depend on ..infrastructure..
..application.. must not depend on ..infrastructure.mapper..
..interfaces.. must not depend on ..infrastructure.mapper..
```

PR 合并前至少需要说明：

- 已运行哪些测试。
- 未运行哪些测试及原因。
- 是否影响数据库、API、权限、部署。

## 15. 开源与依赖规约

- 新增开源依赖前必须确认许可证。
- 优先选择 Apache 2.0、MIT、BSD 等友好许可证。
- GPL、AGPL 类依赖必须单独评审。
- 依赖用途、替代方案和安全风险应记录到技术决策 ADR。
- 禁止提交密钥、token、数据库密码和私有证书。
