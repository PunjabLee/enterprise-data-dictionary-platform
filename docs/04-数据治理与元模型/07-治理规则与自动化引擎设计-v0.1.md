# 治理规则与自动化引擎设计 v0.1

## 1. 设计目标

治理规则与自动化引擎用于支撑平台从“资产展示工具”升级为“治理闭环平台”。其目标是自动发现问题、自动生成任务、自动推动确认、自动支撑审批和自动沉淀治理证据。

## 2. 自动化引擎组成

| 引擎 | 职责 |
| --- | --- |
| Rule Engine | 执行完整性、标准、敏感、分级、影响等级、审批路由等规则 |
| Diff Engine | 比对采集快照、导入数据和已发布资产版本 |
| Task Engine | 生成确认任务、整改任务、认责任务、审批任务 |
| Event Engine | 发布资产变更、采集完成、标准发布、血缘确认等领域事件 |
| Notification Engine | 发送站内通知、邮件、企业微信/钉钉、待办提醒 |
| Report Engine | 生成评审材料、影响分析报告、整改报告和治理统计 |

## 3. 自动化闭环

典型链路：

```text
采集任务完成
-> Diff Engine 识别差异
-> Rule Engine 执行校验
-> Task Engine 生成任务
-> Notification Engine 通知责任人
-> 人工确认或整改
-> 审批或发布
-> Event Engine 发布领域事件
-> Report Engine 更新报告和看板
```

## 4. Rule Engine

### 4.1 规则类型

| 规则类型 | 示例 |
| --- | --- |
| 完整性规则 | 核心表必须有中文名、业务定义、Owner、Steward |
| 标准映射规则 | 核心字段必须引用标准或填写例外说明 |
| 命名规范规则 | 表名、字段名、实体名必须符合命名规则 |
| 敏感识别规则 | 字段名或定义命中手机号、证件号、地址等规则 |
| 分类分级规则 | 根据敏感标签、业务域、资产类型建议分级 |
| 影响等级规则 | 影响核心报表、核心接口、敏感字段时判定高风险 |
| 审批路由规则 | 高敏字段变更必须安全合规会签 |
| 治理评分规则 | 根据完整率、映射率、认领率、确认率计算评分 |

### 4.2 规则属性

| 属性 | 说明 |
| --- | --- |
| 规则编码 | 全局唯一 |
| 规则名称 | 展示名称 |
| 规则类型 | 完整性、标准、敏感、分级、审批路由等 |
| 适用范围 | 业务域、系统、资产类型、字段类型 |
| 触发时机 | 导入后、采集后、提交前、审批前、定时扫描 |
| 规则版本 | 用于追溯执行结果 |
| 启用状态 | 启用、停用 |
| 严重等级 | 提示、警告、阻断 |
| 责任角色 | Owner、Steward、架构师、安全人员 |

### 4.3 规则执行结果

必须记录：

- 规则版本。
- 输入对象。
- 触发时间。
- 命中结果。
- 严重等级。
- 生成任务。
- 处理状态。

## 5. Diff Engine

### 5.1 差异来源

- 数据库采集快照。
- Excel 导入批次。
- API 导入数据。
- 外部系统同步。
- 手工维护变更。

### 5.2 差异类型

| 类型 | 说明 |
| --- | --- |
| 新增 | 源端出现新资产 |
| 修改 | 源端属性与平台版本不一致 |
| 删除 | 源端资产消失 |
| 冲突 | 技术事实与治理定义冲突 |
| 缺失 | 治理必填属性缺失 |

### 5.3 差异处理动作

- 确认。
- 忽略。
- 修正。
- 发起变更审批。
- 生成整改任务。
- 发布新版本。

## 6. Task Engine

任务类型：

| 类型 | 触发来源 |
| --- | --- |
| 确认任务 | 采集结果、血缘解析、导入批次 |
| 整改任务 | 规则校验失败、审批驳回、治理缺口 |
| 认责任务 | Owner 或 Steward 缺失 |
| 审批任务 | 架构评审、模型变更、标准发布 |
| 例外任务 | 标准豁免、架构例外、临时不整改 |

任务字段：

- 任务编号。
- 任务类型。
- 关联资产。
- 触发来源。
- 责任人。
- 截止时间。
- 优先级。
- 状态。
- 处理意见。

## 7. Event Engine

领域事件：

| 事件 | 说明 |
| --- | --- |
| AssetPublished | 资产发布 |
| AssetChanged | 资产变更 |
| StandardPublished | 标准发布 |
| ModelChangeApproved | 模型变更审批通过 |
| LineageConfirmed | 血缘确认 |
| ImportBatchCompleted | 导入批次完成 |
| CollectTaskCompleted | 采集任务完成 |
| ImpactAnalysisCompleted | 影响分析完成 |
| IssueCreated | 治理问题生成 |

实现策略：

- 首期可用本地事件。
- 事件接口统一抽象为 `DomainEventPublisher`。
- 后续可适配 RocketMQ 或 Kafka。
- 事件消费必须幂等。

## 8. Notification Engine

通知渠道：

- 站内通知。
- 邮件。
- 企业微信。
- 钉钉。
- 外部待办平台。

通知触发：

- 审批待办。
- 认责任务。
- 血缘确认。
- 整改任务。
- 采集异常。
- 订阅资产变更。
- 例外到期。

通知要求：

- 支持已读/未读。
- 支持按角色和责任人发送。
- 支持通知模板。
- 支持发送失败重试。

## 9. Report Engine

自动生成：

- 架构评审材料。
- 模型变更影响报告。
- 标准整改报告。
- 资产治理质量报告。
- 血缘确认报告。
- 采集差异报告。

报告必须可追溯：

- 生成时间。
- 数据范围。
- 规则版本。
- 关联任务。
- 关联审批单。

## 10. 自动化边界

禁止自动化绕过人工责任：

- 敏感识别可以自动建议，但必须人工确认。
- 分类分级可以自动建议，但必须安全角色确认。
- 血缘可以自动生成，但正式使用前必须确认。
- 影响等级可以自动判定，但审批结论不能自动通过。
- 采集差异不能直接覆盖已发布业务定义。

## 11. 首期实现建议

P0/P1 分阶段实现：

| 阶段 | 实现 |
| --- | --- |
| P0 | 导入校验、采集差异、完整性规则、标准映射规则、任务生成 |
| P1 | 敏感识别、分类分级建议、影响等级规则、通知中心 |
| P2 | 审批路由规则、治理评分、报告生成、外部通知集成 |
