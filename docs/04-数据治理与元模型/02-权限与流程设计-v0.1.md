# 权限与流程设计 v0.1

## 1. 设计目标

权限与流程设计需要保证平台不仅能展示数据资产，还能把架构治理、数据标准、模型变更、血缘确认和责任认领真正嵌入日常工作。

## 2. 权限模型

### 2.1 权限类型

| 类型 | 说明 |
| --- | --- |
| 菜单权限 | 控制用户能访问哪些功能模块 |
| 操作权限 | 控制新增、编辑、删除、发布、审批、导出 |
| 数据范围权限 | 按业务域、系统、组织、资产类型限制访问范围 |
| 敏感字段权限 | 控制敏感字段定义、分类分级和安全属性可见性 |
| 流程权限 | 控制审批、驳回、转办、撤回、例外处理 |

### 2.2 默认角色

| 角色 | 关键权限 |
| --- | --- |
| 平台管理员 | 系统配置、用户角色、数据源、采集任务 |
| 企业架构师 | 架构评审、系统依赖、架构视图、合规检查 |
| 数据架构师 | 模型管理、模型评审、数据域维护 |
| 数据治理管理员 | 标准术语、质量规则、责任体系、治理看板 |
| Data Steward | 字典维护、标准映射、血缘确认、责任维护 |
| 系统 Owner | 本系统资产确认、接口维护、模型变更提交 |
| 数据 Owner | 业务定义确认、指标口径审批、数据责任确认 |
| 安全合规 | 分类分级、敏感字段、导出审计 |
| 普通用户 | 已授权资产查询、定义查看、血缘查看 |

## 3. 流程清单

### 3.1 系统建设架构评审

触发场景：

- 新建系统。
- 核心系统重大改造。
- 新增跨系统数据交换。

检查项：

- 系统和业务域是否登记。
- 核心数据对象是否识别。
- 是否引用数据标准。
- 是否复用已有模型。
- 是否识别上下游影响。
- 是否完成分类分级。

### 3.2 数据模型变更审批

触发场景：

- 新增、修改、删除核心表或字段。
- 调整逻辑实体或逻辑属性。
- 修改接口字段或指标口径。

流程节点：

1. 提交变更。
2. 自动生成差异和影响分析。
3. 数据架构师评审。
4. 数据 Owner 确认业务影响。
5. 企业架构师确认架构合规。
6. 发布新版本。

### 3.3 数据标准发布流程

触发场景：

- 新增标准。
- 修订标准。
- 废止标准。

流程节点：

1. 数据治理团队起草。
2. 数据 Owner 和数据架构师评审。
3. 安全合规按需会签。
4. 发布标准。
5. 生成引用和整改清单。

### 3.4 血缘确认流程

触发场景：

- 自动解析产生新血缘。
- 手工补录血缘。
- 影响分析发现血缘断点。

确认规则：

- 表级血缘由系统 Owner 或数据平台负责人确认。
- 关键字段级血缘由 Data Steward 和数据平台负责人确认。
- 已确认血缘才能作为正式影响分析依据。

### 3.5 数据资产认责流程

触发场景：

- 新资产采集入库。
- 资产责任人缺失。
- 组织或系统负责人变更。

认责要求：

- 核心系统必须有系统 Owner。
- 核心表必须有数据 Owner 和 Data Steward。
- 核心报表和指标必须有业务责任人。

## 4. 审计要求

必须记录审计日志的操作：

- 登录、登出。
- 新增、修改、删除、发布核心资产。
- 审批、驳回、撤回、转办。
- 导入、导出。
- 权限变更。
- 敏感字段查看。
- 采集连接配置变更。

审计字段：

- 操作人。
- 操作时间。
- 操作对象。
- 操作类型。
- 变更前后摘要。
- 请求来源。
- TraceId。
