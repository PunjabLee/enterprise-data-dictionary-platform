# 业务架构详细设计 v0.1

## 1. 设计目标

本文档把业务架构规范落到可实施的业务流程、能力拆分、输入输出、角色协作和验收场景。本文档面向 Product/Architecture Owner、开发 Agent、测试验收和 PR Steward 使用。

设计目标：

- 将 `08-业务与应用架构视图-v0.1.md` 中的能力地图和价值流拆成可交付 Work Package。
- 明确每个核心治理流程的触发条件、输入、输出、角色、状态和异常处理。
- 明确业务能力到应用上下文、数据对象、API/事件和测试验收的映射。

## 3. 核心流程详细设计

### 3.1 资产导入与确认

| 项目 | 设计 |
| --- | --- |
| 触发条件 | Excel 导入、Open API 导入、采集任务完成 |
| 输入 | 导入批次、模板版本、资产数据、字段数据、关系数据、责任信息 |
| 输出 | ImportBatch、ImportError、Asset、AssetVersion、Diff、Task |
| 主角色 | 平台管理员、系统 Owner、Data Steward |
| 自动规则 | 模板校验、唯一性校验、必填校验、标准映射建议、敏感字段识别建议 |
| 完成条件 | 批次状态为成功或部分成功，关键错误已关闭，确认任务已生成或处理 |

状态建议：

```text
uploaded -> validating -> validated -> importing -> imported -> confirmed
uploaded -> validating -> failed
imported -> partially_confirmed -> confirmed
```

异常处理：

- 模板版本不匹配时拒绝导入，并生成错误明细。
- 同一资产编码冲突时进入差异确认，不覆盖已发布版本。
- 缺少 Owner 或 Steward 时生成责任认领任务。

### 3.2 系统建设架构评审

| 项目 | 设计 |
| --- | --- |
| 触发条件 | 新建系统、系统改造、接口新增、报表新增、指标新增 |
| 输入 | 系统信息、业务域、核心数据对象、接口、模型、标准引用、影响范围、安全分级 |
| 输出 | ReviewCase、ReviewCheckItem、ApprovalTask、AuditLog |
| 主角色 | 系统 Owner、企业架构师、数据架构师、安全合规人员 |
| 自动规则 | 标准引用检查、核心对象完整性检查、重复建设提示、敏感等级会签、影响范围提示 |
| 完成条件 | 必要检查项关闭，审批通过或有明确驳回/豁免结论 |

检查项至少包括：

- 业务域、系统 Owner、数据 Owner 是否明确。
- 核心数据对象和接口是否登记。
- 是否引用已有标准、术语和主数据。
- 是否存在下游影响和高敏数据。
- 是否需要安全合规会签。

### 3.3 模型变更审批

| 项目 | 设计 |
| --- | --- |
| 触发条件 | 概念模型、逻辑模型、物理模型、表字段或接口字段变更 |
| 输入 | 变更前版本、变更后草稿、变更说明、影响分析 |
| 输出 | ChangeRequest、ImpactAnalysis、ApprovalTask、AssetVersion |
| 主角色 | 系统 Owner、数据架构师、Data Steward、企业架构师 |
| 自动规则 | 字段删除影响、类型变化影响、标准偏离、敏感等级变化、血缘下游影响 |
| 完成条件 | 审批通过后发布新版本，旧版本进入历史状态 |

约束：

- 已发布模型不得直接覆盖。
- 删除字段、变更主键、变更敏感等级必须进行影响分析。
- 影响分析报告必须区分已确认血缘和待确认血缘。

### 3.4 标准与术语发布

| 项目 | 设计 |
| --- | --- |
| 触发条件 | 新增术语、修订定义、新增标准、变更指标口径 |
| 输入 | 术语/标准草稿、业务域、责任人、引用对象、冲突检查结果 |
| 输出 | BusinessTerm、DataStandard、TermRelation、TermReference、ApprovalTask |
| 主角色 | 数据治理管理员、数据 Owner、数据架构师 |
| 自动规则 | 同名冲突、跨域冲突、缩写冲突、引用影响分析 |
| 完成条件 | 审批通过并发布，可被资产、字段、指标、模型引用 |

### 3.5 血缘确认与影响分析

| 项目 | 设计 |
| --- | --- |
| 触发条件 | 自动解析、手工补录、导入、模型变更、报表溯源 |
| 输入 | 上游资产、下游资产、关系类型、粒度、来源、可信度 |
| 输出 | LineageRelation、ImpactAnalysis、Task、AuditLog |
| 主角色 | Data Steward、系统 Owner、数据平台负责人 |
| 自动规则 | 低可信度标记、断点提示、循环引用提示、关键字段人工确认 |
| 完成条件 | 正式血缘进入已确认状态；待确认血缘可展示但不得默认作为正式影响依据 |

影响分析默认规则：

- 默认只使用已确认血缘。
- 用户可选择包含待确认血缘，但报告必须标记可信度。
- 关键字段级血缘必须人工确认后进入正式报告。

### 3.6 责任认领、整改与豁免

| 项目 | 设计 |
| --- | --- |
| 触发条件 | 无 Owner、无 Steward、规则命中、质量问题、安全风险、长期未确认血缘 |
| 输入 | 问题对象、规则结果、责任建议、整改要求、期限 |
| 输出 | Issue、Task、Responsibility、Waiver、AuditLog |
| 主角色 | 数据 Owner、系统 Owner、Data Steward、安全合规、治理管理员 |
| 自动规则 | 逾期提醒、复发标记、整改验收、豁免到期复审 |
| 完成条件 | 问题关闭、整改验收通过或有效豁免生效 |

## 5. 验收场景矩阵

| 场景 | 通过标准 |
| --- | --- |
| 系统新建评审 | 可创建评审单，自动生成检查项，完成架构师审批并留痕 |
| 模型变更 | 可提交字段变更，查看影响分析，审批后发布新版本 |
| 标准引用 | 字段可关联标准和术语，冲突可被识别 |
| 血缘确认 | 待确认血缘可确认或驳回，正式影响报告只默认使用已确认血缘 |
| 责任认领 | 无责任资产生成任务，责任人确认后资产责任信息更新 |
| 敏感导出 | L3 及以上资产导出被权限控制并记录审计 |
| 整改闭环 | 规则命中生成问题，整改后可验收关闭或形成豁免 |

## 6. 决策与变更控制

- 业务能力、流程、验收口径变化必须进入产品需求、架构文档或 ADR，不得只保留在聊天记录。
- 新增业务流程必须明确触发、输入、输出、角色、状态、审计和验收。
- 不允许开发 Agent 自行扩大首期范围。
- 涉及 API、数据库、权限、安全、部署的业务变更必须同步影响分析。
