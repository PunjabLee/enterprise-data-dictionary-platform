# 数据架构详细设计 v0.1

## 1. 设计目标

本文档细化数据架构规范，明确数据域、元模型对象、关系矩阵、物理落表、状态版本、血缘、标准术语、安全、质量整改、审计证据和集成同步口径。

## 2. 数据域模型设计

### 2.1 数据域对象

| 对象 | 字段 | 说明 |
| --- | --- | --- |
| BusinessDomain | code、name、description、ownerOrg、status | 企业业务范围 |
| DataDomain | code、name、businessDomainCode、dataOwnerId、stewardId、status | 数据治理范围 |
| SubjectArea | code、name、dataDomainCode、ownerId、description、status | 主题域 |
| SystemGroup | code、name、ownerId、lifecycle、status | 系统群或应用域 |

编码规则：

- code 使用英文小写、数字和连字符。
- code 在租户内唯一。
- 名称可中文，但不得作为唯一标识。
- 废弃数据域不得直接删除，必须保留历史引用。

### 2.2 与权限关系

数据域用于：

- 资产检索过滤。
- 数据范围权限。
- 任务和审批路由。
- 治理看板统计。
- 标准术语适用范围。

## 6. 状态与版本详细设计

### 6.1 资产版本

发布流程：

```text
draft -> pending_confirm -> published -> deprecated
```

版本生成规则：

- 首次发布生成 version 1。
- 影响业务定义、字段结构、标准引用、敏感等级、责任关系的变更生成新版本。
- 纯标签或展示字段变更可不生成业务版本，但必须审计。

### 6.2 标准和术语版本

版本生成规则：

- 定义、适用范围、责任人、状态变化生成新版本。
- 同义词、缩写、引用变化记录审计，是否生成版本由业务规则决定。
- 已发布术语和标准不得原地覆盖定义。

### 6.3 审批与业务对象状态

审批状态不替代业务对象状态：

```text
ApprovalCase approved -> 触发 Model published
ApprovalCase rejected -> 业务对象保持 draft 或 pending 状态
```

## 7. 血缘详细设计

血缘关系字段：

| 字段 | 说明 |
| --- | --- |
| from_asset_id | 上游资产 |
| to_asset_id | 下游资产 |
| relation_type | sync、transform、reference、derive、consume、publish |
| lineage_level | system、interface、table、report、key_field |
| source | collect、import、manual、external |
| confidence | high、medium、low |
| confirm_status | pending、confirmed、rejected、needs_fix |
| confirm_by | 确认人 |
| confirm_at | 确认时间 |
| version | 版本 |

影响分析算法首期口径：

- 默认按下游方向查询影响范围。
- 支持限制深度和资产类型。
- 默认只使用 `confirm_status = confirmed`。
- 可选包含 pending/low confidence，但报告必须标识。
- 循环路径需要去重，报告中展示截断提示。

## 10. 质量与整改数据设计

问题对象字段：

| 字段 | 说明 |
| --- | --- |
| issue_type | completeness、standard_missing、lineage_break、security_risk、quality_result |
| source | rule、import、lineage、dq_platform、manual |
| object_type | 资产类型 |
| object_id | 资产 ID |
| severity | low、medium、high、critical |
| owner_id | 责任人 |
| status | open、processing、resolved、verified、closed、waived |
| due_at | 截止时间 |

整改闭环：

```text
open -> processing -> resolved -> verified -> closed
open -> waived
resolved -> reopened
```

看板指标：

- 问题打开数。
- 逾期数。
- 关闭率。
- 豁免率。
- 复发率。
- 按业务域/数据域/系统分布。

## 11. 审计与治理证据设计

治理证据包括：

- 导入批次和错误明细。
- 采集快照。
- 差异识别结果。
- 规则命中结果。
- 审批意见和附件。
- 资产版本快照。
- 血缘确认记录。
- 整改验收记录。
- 豁免记录。
- 操作审计日志。

证据保留规则：

- 审批、审计、发布版本不得物理删除。
- 导入原始文件可按保留期清理，但批次和摘要必须保留。
- 敏感文件必须限权访问并记录下载审计。

## 12. 集成同步详细设计

| 来源 | 入站对象 | 幂等键 | 覆盖规则 |
| --- | --- | --- | --- |
| IAM | 用户、组织、账号状态 | externalUserId | 更新身份状态，不覆盖本地角色策略 |
| CMDB | 系统、应用、负责人、生命周期 | cmdbSystemCode | 更新运行事实，治理定义需确认 |
| Database | 库、Schema、表、字段 | datasourceId + objectPath | 生成技术事实和差异，不覆盖发布定义 |
| ETL/BI | 任务、报表、数据集、依赖 | externalObjectId | 生成血缘候选，需确认 |
| DevOps | 变更单、发布单 | changeId | 关联评审证据 |
| DQ Platform | 质量规则、质量结果 | dqResultId | 生成或更新问题，不替代 DQ 执行 |

失败处理：

- 入站数据校验失败进入错误明细。
- 幂等冲突进入差异处理。
- 外部系统不可用时记录失败和重试，不阻断平台核心功能。

## 13. 数据验收场景

- 数据域可作为权限和筛选条件使用。
- 资产发布生成版本快照。
- 字段可引用术语和标准。
- 模型变更生成影响分析。
- 已确认血缘进入正式影响分析。
- 敏感字段导出被审计。
- 质量问题可整改、验收、关闭或豁免。
- 外部 CMDB/数据库采集差异不会直接覆盖已发布定义。
