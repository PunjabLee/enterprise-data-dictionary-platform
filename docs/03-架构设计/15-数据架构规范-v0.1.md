# 数据架构规范 v0.1

## 1. 定位

本文档定义平台数据架构规范，用于统一数据域、元模型、物理模型、状态版本、血缘、标准术语、安全分级、质量整改、审计证据和集成数据边界。

本文档不替代 `04-数据治理与元模型` 下的专题设计，而是作为业务、应用、技术、数据四类架构之间的数据约束总控。

## 2. 数据架构原则

| 原则 | 规范 |
| --- | --- |
| 元模型驱动 | 资产类型、属性、关系类型优先通过元模型表达 |
| 事实与定义分离 | 采集结果是技术事实，发布资产是治理定义 |
| 状态版本独立 | 已发布对象不得原地覆盖，变更必须有版本和状态 |
| 关系可追溯 | 包含、映射、引用、依赖、血缘、责任、影响必须可建边 |
| 安全默认内建 | 分类分级、敏感字段、数据范围和导出审计是基础能力 |
| 审计证据完整 | 导入、采集、审批、发布、导出、整改、豁免必须留痕 |
| 可扩展但不滥用 | 新资产类型优先配置，核心对象可独立建模以保证性能和语义清晰 |

## 3. 数据域与主题域

数据域不应仅作为自由文本。首期应作为元模型资产类型管理，并在核心资产表保留索引用字段。

| 概念 | 说明 | 关键属性 |
| --- | --- | --- |
| BusinessDomain | 企业业务范围，例如客户、产品、交易、风险 | code、name、ownerOrg、status |
| DataDomain | 数据治理范围，例如客户数据域、财务数据域 | code、name、businessDomainCode、dataOwner |
| SubjectArea | 主题域，例如客户基本信息、客户标签 | code、name、dataDomainCode、steward |
| SystemGroup | 系统群或应用域 | code、name、owner、lifecycle |

资产表应保留：

- `business_domain_code`。
- `data_domain_code`。
- `subject_area_code`。
- `system_code`。
- `owner_user_id` 或责任关系。

## 4. 元模型对象规范

首期元模型对象范围：

| 类型 | 说明 |
| --- | --- |
| System | 系统、应用、服务 |
| Database | 数据库实例或逻辑库 |
| Schema | 数据库 Schema |
| Table | 物理表 |
| Field | 表字段或关键字段 |
| Interface | API、文件接口、消息接口 |
| ETLJob | ETL、调度任务、数据同步任务 |
| Report | 报表、数据集、看板 |
| Metric | 指标、口径 |
| ConceptModel | 概念模型 |
| LogicalModel | 逻辑模型 |
| PhysicalModel | 物理模型 |
| BusinessTerm | 业务术语 |
| DataStandard | 数据标准 |
| Issue | 问题台账 |
| Task | 治理任务 |
| ApprovalCase | 审批或评审单 |

## 5. 关系类型规范

| 关系 | 用途 | 示例 |
| --- | --- | --- |
| contains | 层级包含 | System contains Database |
| maps_to | 模型或标准映射 | LogicalModel maps_to PhysicalModel |
| references | 引用 | Field references DataStandard |
| depends_on | 依赖 | Report depends_on Table |
| lineage | 血缘 | Table lineage Table |
| owns | 责任 | DataOwner owns Asset |
| impacts | 影响 | Field impacts Report |
| rectifies | 整改 | Issue rectifies Asset |
| classifies | 分类 | Field classifies SensitiveCategory |

关系约束：

- 所有关系必须包含来源、状态、版本或有效期。
- 血缘关系必须包含粒度、来源、可信度和确认状态。
- 责任关系必须可用于任务路由和数据范围控制。

## 6. 元模型到物理模型规范

| 元模型对象 | 主表 | 扩展属性 | 关系表 | 版本 |
| --- | --- | --- | --- | --- |
| 通用资产 | asset | asset_attribute_value | asset_relation | asset_version |
| 术语 | business_term | term_version | term_relation、term_reference | term_version |
| 标准 | data_standard | standard_version | standard_reference | standard_version |
| 血缘 | lineage_relation | lineage_attribute | lineage_relation | lineage_version 或 version 字段 |
| 审批 | approval_case | approval_task | business_key 引用 | process 状态 |
| 问题 | issue | issue_record | issue_asset_relation | issue 状态 |
| 任务 | task | task_record | task_object_relation | task 状态 |
| 审计 | sys_audit_log | before/after JSON | object_type/object_id | 不覆盖 |

物理模型通用字段：

- `id`。
- `tenant_id`，首期默认 `default`。
- `created_by`、`created_at`。
- `updated_by`、`updated_at`。
- `deleted`。
- `version`。

## 7. 状态与版本规范

对象状态必须和审批流程分离，但可以由审批结果触发状态变化。

| 对象 | 状态示例 | 版本规则 |
| --- | --- | --- |
| 资产 | draft、pending_confirm、published、deprecated | 发布生成新版本 |
| 模型 | draft、reviewing、published、archived | 审批通过后发布 |
| 标准 | draft、reviewing、published、deprecated | 发布后不可原地覆盖 |
| 术语 | draft、reviewing、published、deprecated | 定义变化生成版本 |
| 血缘 | pending、confirmed、rejected、deprecated | 确认状态变化留痕 |
| 审批单 | draft、processing、approved、rejected、cancelled | 不覆盖审批意见 |
| 问题 | open、processing、resolved、verified、closed、waived | 整改和验收留痕 |

## 8. 血缘与影响分析规范

血缘粒度：

- 系统级。
- 接口级。
- 表级。
- 报表级。
- 关键字段级。

首期不承诺全量字段级血缘。

血缘来源：

- 自动采集解析。
- Excel/API 导入。
- 手工补录。
- 外部 ETL/BI/调度平台同步。

可信度规则：

- `confirmed` 血缘可作为正式影响分析依据。
- `pending` 或低可信血缘可展示，但报告必须显式标识。
- 关键字段级血缘必须人工确认。

## 9. 标准术语与字典规范

- `BusinessTerm` 保持独立领域对象，同时同步到统一资产索引用于检索。
- `TermReference` 连接术语与字段、指标、模型、报表、标准。
- `StandardReference` 连接标准与资产、字段、模型或接口。
- 字典字段可引用术语和标准，但不得复制术语定义作为事实源。
- 术语冲突、标准冲突、跨域定义冲突必须进入治理流程。

## 10. 数据安全规范

| 安全能力 | 规范 |
| --- | --- |
| 分类 | 个人信息、企业敏感、财务、经营、技术、公开等 |
| 分级 | L1 到 L4，L3 及以上导出必须审计 |
| 敏感识别 | 可自动建议，但高敏必须人工确认 |
| 展示控制 | 按权限决定展示、脱敏或隐藏 |
| 导出控制 | 根据权限、分级、审批和审计控制 |
| 数据范围 | 按业务域、系统、组织、资产类型控制 |

## 11. 数据质量与整改规范

平台不替代 DQ 执行引擎。平台负责：

- 接入质量规则和质量结果。
- 将质量问题映射到资产、字段、Owner 和任务。
- 形成整改、验收、豁免和复发记录。
- 形成治理看板指标。

问题来源：

- 导入校验。
- 元数据完整性规则。
- 标准映射规则。
- 敏感字段未分级。
- 血缘断点或长期未确认。
- 外部 DQ 平台结果。

## 12. 集成数据边界

| 外部系统 | 权威边界 | 平台边界 |
| --- | --- | --- |
| IAM/OIDC/LDAP/AD | 身份、组织、账号状态 | 本地用户映射、权限、数据范围 |
| CMDB | 系统运行事实、生命周期、负责人 | 治理定义、架构评审证据 |
| 数据库 | 技术元数据事实 | 治理定义、版本、标准、责任 |
| ETL/调度/BI | 任务、依赖、报表、数据集事实 | 血缘治理、影响分析、确认流程 |
| DevOps | 变更单、发布单、流水线事实 | 评审结果和上线前置证据 |
| 数据质量平台 | 质量执行和质量结果 | 整改闭环、责任追踪、治理看板 |

## 13. 禁止事项

- 禁止采集业务明细数据作为平台主目标。
- 禁止采集账号拥有写权限。
- 禁止采集差异直接覆盖已发布治理定义。
- 禁止已发布对象原地覆盖。
- 禁止把 OpenSearch 索引作为事实源。
- 禁止质量平台执行职责被本平台隐式接管。
