# 业务架构规范 v0.1

## 1. 定位

本文档定义企业数据架构治理与元数据运营平台的业务架构规范，用于统一业务能力、参与方、价值流、治理流程、验收口径和跨架构映射。

本规范基于 `08-业务与应用架构视图-v0.1.md`，不改变既有 ADR 和首期 MVP 范围。首期目标是跑通治理闭环，而不是建设全量 TOGAF 制品库或替代源系统、ETL、BI、数据质量平台。

## 2. 业务架构原则

| 原则 | 规范 |
| --- | --- |
| 治理闭环优先 | 业务能力必须支撑发现、校验、任务、确认、审批、发布、消费和度量的闭环 |
| 元模型统一 | 业务对象必须能映射到统一元模型、状态、版本、责任和审计证据 |
| 分域推广 | 平台采用统一框架，但按业务域、系统群和数据域分批纳管 |
| 自动化加人工治理 | 技术事实优先自动采集，业务定义、责任、标准映射、确认和例外必须可人工治理 |
| 权责清晰 | 每项治理动作必须明确 Owner、Steward、架构师、安全合规或平台管理员职责 |
| 证据可追溯 | 评审、审批、发布、导入、变更、导出和豁免必须留存审计证据 |

## 3. 业务能力规范

| 能力域 | 核心能力 | 首期优先级 | 业务结果 |
| --- | --- | --- | --- |
| 资产运营 | 资产目录、数据字典、元模型管理、状态版本 | P0 | 形成可检索、可维护、可追溯的资产事实和治理定义 |
| 语义与标准 | 业务术语、数据标准、指标口径、标准映射 | P0 | 统一业务定义和标准引用 |
| 治理决策 | 架构评审、模型变更审批、标准发布、例外豁免 | P0 | 将治理要求嵌入系统建设和变更流程 |
| 血缘影响 | 血缘维护、血缘确认、影响分析、报表溯源 | P0 | 支撑变更影响评估和责任追踪 |
| 自动化治理 | 采集导入、差异识别、规则校验、任务通知 | P0 | 减少人工盘点，驱动整改和确认任务 |
| 安全审计 | RBAC、数据范围、敏感字段、操作审计 | P0 | 控制访问、导出和敏感信息暴露 |
| 运营度量 | 完整率、标准映射率、认领率、血缘覆盖率、整改闭环率 | P1 | 衡量治理质量和推广进展 |

## 4. 参与方与职责

| 参与方 | 核心职责 | 典型产出 |
| --- | --- | --- |
| 平台管理员 | 用户、角色、权限、数据源、模板和系统配置 | 权限配置、数据源配置、审计记录 |
| 企业架构师 | 架构原则、系统依赖、架构评审、例外确认 | 架构评审意见、例外记录 |
| 数据架构师 | 数据域、模型设计、模型评审、标准引用确认 | 模型评审意见、标准引用结果 |
| 数据治理管理员 | 术语标准、治理规则、责任体系、治理看板 | 规则配置、治理指标 |
| Data Steward | 字典维护、定义补齐、血缘确认、整改处理 | 字段定义、血缘确认、整改结果 |
| 系统 Owner | 系统资产确认、接口维护、模型变更提交 | 资产确认、变更申请 |
| 数据 Owner | 业务定义、指标口径、授权确认、数据责任 | 定义确认、授权记录 |
| 安全合规人员 | 分类分级、敏感字段、导出控制、审计要求 | 安全会签、导出审批 |
| 普通用户 | 查询资产定义、口径、来源、血缘和责任人 | 查询和订阅记录 |
| 外部系统 | 提供身份、运行事实、采集来源、流程或质量结果 | 集成数据、事件、回执 |

## 5. 业务价值流规范

业务价值流固定为：

```text
发现变化 -> 进入元模型 -> 差异识别 -> 规则校验 -> 任务生成
-> 人工治理 -> 审批发布 -> 消费使用 -> 运营度量 -> 差异识别
```

价值流约束：

- 采集或导入结果是技术事实，不得直接覆盖已发布治理定义。
- 差异识别必须区分新增、修改、删除、冲突、缺失和低可信。
- 规则校验至少覆盖完整性、标准映射、敏感分级、影响等级和责任归属。
- 任务生成必须落到明确角色和对象，支持确认、整改、认责、审批和豁免。
- 审批发布必须形成版本、状态、审批记录和审计日志。
- 消费使用必须区分正式数据和待确认数据，特别是血缘和敏感字段。

## 6. 核心治理流程规范

| 流程 | 触发 | 关键参与方 | 完成条件 |
| --- | --- | --- | --- |
| 资产导入确认 | Excel/API/采集批次进入平台 | 平台管理员、系统 Owner、Data Steward | 导入批次完成校验，关键资产完成确认或生成整改任务 |
| 系统建设架构评审 | 新建系统、系统改造、接口新增 | 系统 Owner、企业架构师、数据架构师、安全合规 | 评审项完成，风险与例外有结论，形成评审证据 |
| 模型变更审批 | 表、字段、接口字段、模型结构变更 | 系统 Owner、数据架构师、Data Steward | 影响分析完成，审批通过后发布新版本 |
| 标准/术语发布 | 新增或变更业务术语、数据标准、指标口径 | 数据治理管理员、数据 Owner、数据架构师 | 冲突检查完成，审批通过，引用关系更新 |
| 血缘确认 | 自动解析、手工补录、影响分析发现断点 | Data Steward、系统 Owner、数据平台负责人 | 血缘关系进入已确认或需修正状态 |
| 责任认领 | 无 Owner/Steward 或责任变更 | 数据 Owner、系统 Owner、治理管理员 | 责任关系完成确认并可用于权限和任务路由 |
| 问题整改 | 规则命中、质量问题、安全风险、血缘断点 | 责任人、Data Steward、安全合规 | 整改验收通过或形成有效豁免 |
| 例外豁免 | 标准偏离、流程例外、短期无法整改 | 企业架构师、数据治理管理员、安全合规 | 豁免范围、期限、风险和复审机制明确 |

## 7. 首期边界

首期必须跑通：

- 资产导入、资产目录、数据字典和元模型管理。
- 标准术语、基础血缘、影响分析。
- 架构评审、模型变更、责任认领。
- RBAC、数据范围、敏感字段控制和操作审计。
- 差异识别、规则校验、任务通知和整改闭环的最小链路。

首期不承诺：

- 微服务拆分。
- 复杂 BPMN 建模和自动审批通过。
- 全量字段级血缘。
- 全套 TOGAF 制品库。
- 全量 ETL/BI 适配。
- 替代源系统、数据仓库、ETL、BI 或数据质量执行平台。

## 8. 跨架构映射规范

| 业务能力 | 应用上下文 | 技术支撑 | 数据对象 |
| --- | --- | --- | --- |
| 资产导入 | collector、metadata、automation | Excel/API、批次任务、本地事件 | ImportBatch、Asset、AssetVersion、Diff |
| 资产目录 | catalog、metadata、security | REST API、PostgreSQL 索引、数据范围 | Asset、AssetAttributeValue、Tag |
| 数据字典 | dictionary、metadata、glossary、standard | 应用服务、权限审计 | FieldDictionary、TermReference、StandardReference |
| 标准术语 | glossary、standard、governance | 审批任务、冲突检测 | BusinessTerm、DataStandard、TermRelation |
| 血缘影响 | lineage、metadata、collector | 图查询、按需展开、可信度标记 | LineageRelation、ImpactAnalysis |
| 治理审批 | governance、automation、security | 轻量流程、任务中心、审计 | ReviewCase、ApprovalTask、Responsibility |
| 安全审计 | system/security、integration | RBAC、数据范围、审计日志 | User、Role、Permission、AuditLog |

## 9. 验收口径

业务架构验收以闭环为准，不以资产数量为唯一标准。首期至少证明：

- 系统建设场景可完成架构评审闭环。
- 模型变更场景可完成影响分析、审批和发布闭环。
- 标准术语可发布并被资产、字段、指标或模型引用。
- 血缘可确认，正式影响分析默认使用已确认血缘。
- 无责任资产可生成认责任务并关闭。
- 敏感字段查看、导出和分类分级变更可审计。
- 采集差异或质量问题可进入整改、验收或豁免流程。
