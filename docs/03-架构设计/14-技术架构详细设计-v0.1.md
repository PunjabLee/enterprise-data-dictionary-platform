# 技术架构详细设计 v0.1

## 1. 设计目标

本文档细化技术架构规范，明确工程落地、端口适配、事件机制、搜索、工作流、规则、文件、部署、安全审计和可观测性的实施设计。

## 2. 后端工程设计

建议工程骨架：

```text
platform/backend/
  pom.xml
  metadata-platform/
    pom.xml
    src/main/java/com/example/dictionary/
      DictionaryApplication.java
      common/
      system/
      metadata/
      catalog/
      dictionary/
      modeling/
      glossary/
      standard/
      lineage/
      governance/
      automation/
      collector/
      integration/
    src/main/resources/
      application.yml
      application-local.yml
      db/migration/
    src/test/java/
```

Maven 规范：

- 根 `pom.xml` 统一管理 Java 版本、Spring Boot、MyBatis-Plus、Flyway、PostgreSQL Driver、Redis、测试依赖。
- 业务模块可以先在一个 Spring Boot module 内按包隔离，后续再拆 Maven 子模块。
- 架构测试作为后端测试的一部分运行。

## 3. 前端工程设计

建议工程骨架：

```text
platform/frontend/web-console/
  package.json
  src/
    app/
    router/
    stores/
    api/
      metadata/
      catalog/
      dictionary/
      governance/
    pages/
      workspace/
      catalog/
      dictionary/
      modeling/
      glossary/
      lineage/
      governance/
      import/
      system/
    components/
    styles/
```

前端规范：

- `api/<context>` 与后端 owner context 对齐。
- 路由 meta 包含菜单权限和页面标题。
- 复杂页面的状态保存在 Pinia store 或局部 composable，不跨业务页面共享隐式状态。
- 导出、审批、发布、删除等操作必须显式二次确认，并调用后端审计。

## 4. 事件机制设计

首期使用本地事件机制：

```text
ApplicationService
-> DomainEventPublisher
-> local_domain_event
-> LocalDomainEventDispatcher
-> Subscriber
```

事件表建议字段：

| 字段 | 说明 |
| --- | --- |
| id | 主键 |
| event_id | 全局唯一事件 ID |
| tenant_id | 租户 |
| event_type | 事件类型 |
| producer_context | 生产上下文 |
| aggregate_type | 聚合类型 |
| aggregate_id | 聚合 ID |
| payload | JSON payload |
| event_version | 事件版本 |
| status | pending、published、failed、ignored |
| retry_count | 重试次数 |
| occurred_at | 发生时间 |
| published_at | 发布时间 |
| trace_id | 链路 ID |

处理规则：

- 同一事件多次消费必须幂等。
- 消费失败记录原因和重试次数。
- 对外消息由 integration 适配，内部事件不直接暴露给外部系统。
- 后续接入 MQ 时保持业务代码不变，仅替换 publisher/subscriber adapter。

## 5. 搜索设计

首期搜索路径：

```text
Catalog Query -> PostgreSQL 索引/模糊查询 -> DTO
```

OpenSearch 预留路径：

```text
AssetPublished -> AssetSearchIndexer -> SearchGateway -> OpenSearch
```

搜索规范：

- PostgreSQL 是事实源，搜索索引可重建。
- 搜索 DTO 必须经过权限和数据范围过滤。
- OpenSearch 接入条件：资产/字段规模达到 PostgreSQL 搜索瓶颈、需要复杂分词排序或多维聚合。
- 索引失败不得影响资产发布，但必须产生告警或任务。

## 6. 工作流设计

首期轻量流程对象：

| 对象 | 说明 |
| --- | --- |
| ProcessInstance | 流程实例，关联业务对象 |
| ApprovalTask | 审批任务，关联处理人、状态、意见 |
| ReviewCase | 架构评审单 |
| Responsibility | 责任关系 |
| Waiver | 例外豁免 |

端口：

```text
WorkflowEngine
  start(processType, businessKey, participants)
  approve(taskId, decision, comment)
  reject(taskId, reason)
  cancel(processInstanceId)
```

Flowable 接入条件：

- 出现复杂并行、会签、条件分支、超时升级。
- 企业已有统一流程平台。
- 审批配置需要业务人员自助建模。

## 7. 规则与自动化设计

规则执行入口：

```text
RuleApplicationService
-> RuleEngine
-> RuleExecutor
-> RuleResult
-> Diff/Issue/Task/Notification
```

首期规则类型：

- 完整性规则。
- 标准映射规则。
- 敏感字段识别规则。
- 分类分级建议规则。
- 影响等级规则。
- 审批路由规则。
- 责任缺失规则。

规则约束：

- 规则执行结果必须可追踪。
- 自动化只能建议、标记、生成任务，不自动通过审批。
- 高风险规则命中必须进入人工确认或安全会签。

## 8. 文件与导入导出设计

文件能力：

| 场景 | 首期实现 | 后续适配 |
| --- | --- | --- |
| Excel 导入 | 本地临时文件 + 批次记录 | 对象存储 |
| 错误明细下载 | 生成文件并审计 | 对象存储生命周期 |
| 资产导出 | 权限校验后生成 | 异步导出队列 |
| 敏感附件 | 限权下载 | 加密存储 |

文件约束：

- 文件元数据必须入库。
- 敏感文件下载必须审计。
- 临时文件必须有清理策略。
- 导入批次必须保留模板版本和原始文件引用。

## 9. 安全审计技术设计

安全链路：

```text
AuthenticationFilter -> CurrentUser -> PermissionChecker
-> DataScopeProvider -> ApplicationService -> AuditLogger
```

审计日志字段：

| 字段 | 说明 |
| --- | --- |
| tenant_id | 租户 |
| actor_id | 操作人 |
| actor_name | 操作人名称 |
| action | 操作 |
| object_type | 对象类型 |
| object_id | 对象 ID |
| before_value | 变更前 JSON |
| after_value | 变更后 JSON |
| result | success、failed |
| ip | 来源 IP |
| user_agent | UA |
| trace_id | 链路 ID |
| created_at | 操作时间 |

审计约束：

- 审计日志不得被业务删除。
- 权限变更必须触发权限缓存失效。
- 数据源连接信息必须加密或由环境注入。

## 10. 部署与配置设计

配置分组：

| 配置 | 示例 |
| --- | --- |
| 数据库 | JDBC URL、用户名、连接池 |
| Redis | 地址、密码、超时 |
| 安全 | JWT secret、token 过期时间、OIDC 配置 |
| 文件 | 本地路径、对象存储 endpoint |
| 采集 | 采集超时、最大并发、只读校验 |
| 异步任务 | 线程池、重试次数、批次大小 |
| 可选组件 | OpenSearch、MQ、Flowable profile |

生产上线检查：

- 禁止默认密码。
- Flyway 迁移已执行。
- 审计日志开启。
- PostgreSQL 备份策略已配置。
- 数据源密钥不在仓库。
- 健康检查和日志采集可用。

## 11. 可观测性设计

TraceId 贯穿：

```text
HTTP Request -> ApplicationService -> DomainEvent -> Task -> AuditLog
```

建议指标：

- API 请求量、延迟、错误率。
- DB 连接池使用率和慢查询数量。
- Redis 连接错误。
- 导入批次耗时、错误数。
- 异步任务积压和失败数。
- 事件发布失败数。
- 审批任务超时数。
- 文件存储占用。

日志分层：

- 应用日志。
- 访问日志。
- 任务日志。
- 采集日志。
- 审计日志。
- 集成回调日志。

## 12. 技术验收清单

- 本地 Docker Compose 可启动 PostgreSQL、Redis、后端、前端。
- Flyway 可初始化核心表。
- 后端架构测试可阻断跨层依赖。
- 本地事件发布和消费可验证幂等。
- 导入、发布、审批、导出均产生审计日志。
- OpenSearch、MQ、Flowable 未启用时系统仍可运行。
- 文档中的端口、事件、上下文和数据对象名称与应用架构一致。
