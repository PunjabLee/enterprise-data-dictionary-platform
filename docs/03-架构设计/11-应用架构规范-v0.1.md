# 应用架构规范 v0.1

## 1. 定位

本文档定义平台应用架构的权威规范，用于约束后端模块、前端模块、限界上下文、服务分层、依赖方向、API/事件归属、权限审计和首期交付边界。

本规范继承 `06-应用架构设计-v0.1.md` 与 `08-业务与应用架构视图-v0.1.md`，实施时以本文档的上下文命名和依赖规则为准。

## 2. 应用架构原则

| 原则 | 规范 |
| --- | --- |
| 模块化单体 | 首期一个 Spring Boot 部署单元，内部按限界上下文隔离 |
| DDD 分层 | 每个业务上下文遵循 interfaces、application、domain、infrastructure 分层 |
| 端口优先 | 跨上下文协作优先通过应用端口、领域事件、防腐层或只读查询模型 |
| 依赖单向 | 禁止跨模块直接访问 Mapper、DAO、表或基础设施实现 |
| 业务归属清晰 | API、DTO、事件、任务、审计点必须标注 owner context |
| 可演进 | 保留未来按限界上下文拆分服务的可能，但首期不拆微服务 |

## 3. 权威限界上下文

| 上下文 | 职责 | 主要对象 |
| --- | --- | --- |
| system/security | 用户、组织、角色、权限、数据范围、审计 | User、Org、Role、Permission、AuditLog |
| metadata | 元模型、资产、属性、关系、版本、快照 | MetaType、Asset、AssetRelation、AssetVersion |
| catalog | 资产目录、资产检索、资产详情、订阅关注 | CatalogAsset、Tag、Subscription |
| dictionary | 表字典、字段字典、业务定义维护 | FieldDictionary、DictionaryEntry |
| modeling | 概念模型、逻辑模型、物理模型、模型映射 | ConceptEntity、LogicalEntity、PhysicalModel |
| glossary | 业务术语、术语关系、术语引用、术语冲突 | BusinessTerm、TermRelation、TermReference |
| standard | 数据标准、指标口径、标准引用、发布 | DataStandard、StandardReference |
| lineage | 血缘关系、血缘确认、影响分析 | LineageRelation、ImpactAnalysis |
| governance | 架构评审、审批任务、认责、例外豁免 | ReviewCase、ApprovalTask、Responsibility、Waiver |
| automation | 规则、差异、问题、任务、通知、报告 | Rule、Diff、Issue、Task、Notification、Report |
| collector | 数据源、采集任务、导入批次、采集快照 | DataSource、CollectTask、ImportBatch、Snapshot |
| integration | Open API、Webhook、外部系统防腐、消息适配 | IntegrationApp、Webhook、ExternalMapping |

说明：

- `catalog/dictionary`、`glossary/standard` 可以在架构视图中合并展示，但代码包、API owner 和事件 owner 必须使用上表的权威上下文名。
- `common` 只能放错误码、租户、审计字段、时间、分页、工具等共享基础能力，不作为业务上下文。

## 4. 分层规范

后端上下文统一包结构：

```text
<context>/
  interfaces/
    rest/
    dto/
    assembler/
  application/
    service/
    command/
    query/
    port/
  domain/
    model/
    service/
    event/
    repository/
  infrastructure/
    persistence/
    adapter/
    client/
```

分层职责：

| 层 | 职责 | 禁止事项 |
| --- | --- | --- |
| interfaces | REST API、DTO、请求校验、响应组装 | 编写业务规则、直接访问 Mapper |
| application | 用例编排、事务边界、权限审计、领域事件发布 | 持有基础设施实现细节 |
| domain | 聚合、实体、值对象、领域服务、领域事件 | 依赖 Spring、MyBatis、Redis、MQ、HTTP 客户端 |
| infrastructure | Repository 实现、Mapper、外部客户端、适配器 | 向 domain 反向暴露基础设施类型 |

## 5. 依赖方向

允许依赖：

```text
interfaces -> application -> domain
infrastructure -> application/domain ports
```

跨上下文协作规则：

| 来源 | 目标 | 允许方式 |
| --- | --- | --- |
| catalog、dictionary、modeling、lineage | metadata | 应用端口或只读查询接口 |
| collector | metadata | 导入/采集端口、领域事件 |
| automation | metadata、governance、collector | 事件订阅、规则结果、任务建议 |
| governance | metadata、lineage、standard、glossary | 应用端口、事件消费 |
| integration | 各上下文 | Open API 防腐层、Webhook、集成事件 |
| system/security | 各上下文 | 权限服务、数据范围、审计端口 |

禁止依赖：

- 业务模块直接依赖其他模块的 Mapper、DAO 或数据库表。
- 领域层依赖 infrastructure、Spring、MyBatis、Redis、MQ 或 HTTP 客户端。
- `governance` 与 `automation` 出现编译期循环依赖。
- `integration` 持有核心业务规则。
- `collector` 直接覆盖已发布资产业务定义。

## 6. API、DTO 与事件归属

| 类型 | 规范 |
| --- | --- |
| REST API | 路由前缀必须体现 owner context，例如 `/api/metadata/assets` |
| DTO | DTO 归属 API owner，不跨上下文复用内部 DTO |
| Command | 写操作使用 Command 表达业务意图，由 application 层处理 |
| Query | 查询可使用专用 Query DTO，不反向污染领域模型 |
| Domain Event | 由产生事实的上下文定义，例如 `AssetPublished` 属于 metadata |
| Integration Event | 由 integration 负责对外契约、版本、幂等和适配 |
| Async Task | 任务语义归属 automation/governance，业务对象归属原上下文 |

事件必须包含：

- eventId。
- eventType。
- version。
- occurredAt。
- tenantId。
- producerContext。
- aggregateType。
- aggregateId。
- payload。

## 7. 权限与审计规范

应用服务必须在以下操作记录审计：

- 创建、修改、删除、废弃资产、标准、术语、模型、血缘。
- 发布、审批、驳回、转办、撤回、豁免。
- 导入、导出、下载敏感文件。
- 修改用户、角色、权限、数据范围、敏感等级。
- 修改数据源、采集配置、外部集成配置。

权限控制层次：

| 控制 | 说明 |
| --- | --- |
| 菜单权限 | 控制页面入口 |
| 操作权限 | 控制新增、修改、删除、发布、审批、导出 |
| 数据范围 | 按业务域、系统、组织、资产类型约束可见范围 |
| 敏感字段权限 | 控制敏感定义、分级、安全属性和导出 |
| 流程权限 | 控制审批节点、转办、撤回、终止和豁免 |

## 8. 首期边界

首期必须实现：

- 权威上下文包结构和依赖规则。
- 资产、字典、术语、标准、血缘、治理、权限的最小应用服务。
- REST API、DTO、错误码、权限审计的基础契约。
- 本地领域事件端口和事件记录能力。
- 轻量审批任务模型。

首期不强制实现：

- 微服务拆分。
- Flowable 运行时强依赖。
- RocketMQ/Kafka 强依赖。
- OpenSearch 强依赖。
- 全量字段级血缘。
- 独立图数据库。

## 9. 架构门禁

PR 进入评审前至少满足：

- 不跨上下文直接访问 Mapper 或表。
- domain 层不依赖 infrastructure。
- 新增 API 标注 owner context、权限和审计要求。
- 新增事件标注 producer、consumer、payload、version 和幂等键。
- 新增状态流转对齐状态机与版本规则。
- 涉及安全、权限、数据库、API 的变更在 PR 描述中说明影响。
