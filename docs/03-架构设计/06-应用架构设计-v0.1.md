# 应用架构设计 v0.1

## 1. 设计目标

应用架构用于定义平台的业务模块边界、DDD 限界上下文、模块依赖、应用服务、领域服务、领域事件和跨模块协作方式。首期采用 DDD 分层的模块化单体，后续可按限界上下文拆分服务。

## 2. 应用架构原则

- 模块化单体，一个部署单元，多个清晰业务模块。
- 以限界上下文组织代码，不以数据库表或页面菜单组织业务逻辑。
- 应用层负责编排用例和事务，领域层负责业务规则和领域行为。
- 基础设施层实现 Repository、DAO、Mapper、缓存、消息和外部系统适配。
- 模块之间不直接访问对方 DAO、Mapper 和数据库表。
- 跨模块协作优先通过应用服务接口、领域事件或防腐层完成。

## 3. 限界上下文

| 上下文 | 职责 | 关键对象 |
| --- | --- | --- |
| metadata | 元模型、资产、属性、关系、版本 | MetaType、Asset、AssetRelation、AssetVersion |
| catalog | 资产目录、检索、标签、关注 | CatalogAsset、Tag、Favorite |
| dictionary | 表字典、字段字典、业务定义 | TableDictionary、FieldDictionary |
| modeling | 概念模型、逻辑模型、物理模型 | ConceptEntity、LogicalEntity、PhysicalModel |
| glossary | 业务术语、术语关系、术语引用、术语冲突 | BusinessTerm、TermRelation、TermReference |
| standard | 数据标准、指标口径、编码规则 | DataStandard、StandardReference |
| lineage | 血缘关系、血缘确认、影响分析 | LineageRelation、ImpactAnalysis |
| governance | 架构评审、模型变更、标准发布、责任认领 | ReviewCase、ApprovalTask、Responsibility |
| automation | 规则、差异、任务、通知、报告 | Rule、Diff, GovernanceTask、Notification |
| collector | 数据源、采集任务、导入批次、采集快照 | DataSource、CollectTask、ImportBatch |
| security | 用户、组织、角色、权限、审计 | User、Org、Role、Permission、AuditLog |
| integration | 开放 API、消息、外部系统适配 | IntegrationApp、Webhook、ExternalMapping |

## 4. 模块依赖

推荐依赖方向：

```text
catalog -> metadata
dictionary -> metadata, glossary, standard
modeling -> metadata, glossary, standard
lineage -> metadata
governance -> metadata, glossary, standard, lineage, automation
automation -> metadata, glossary, standard, lineage, governance
collector -> metadata, automation
security -> all modules via permissions/audit ports
integration -> application ports of other modules
```

禁止依赖：

- 业务模块直接依赖其他模块的 Mapper。
- 业务模块直接查询其他模块数据库表。
- 领域层依赖 infrastructure。
- 自动化规则直接修改已发布资产定义。

## 5. 应用服务划分

| 应用服务 | 职责 |
| --- | --- |
| AssetApplicationService | 创建、修改、确认、发布、废弃资产 |
| DictionaryApplicationService | 维护字段业务定义、标准映射、术语引用 |
| BusinessGlossaryApplicationService | 维护术语、术语关系、术语审批、术语影响 |
| StandardApplicationService | 标准创建、修订、发布、引用检查 |
| ModelApplicationService | 模型维护、模型映射、模型变更 |
| LineageApplicationService | 血缘维护、确认、路径查询 |
| ImpactAnalysisApplicationService | 影响分析任务发起、结果查询 |
| GovernanceApplicationService | 架构评审、审批、责任认领、例外申请 |
| AutomationApplicationService | 规则执行、差异处理、任务生成、报告生成 |
| CollectorApplicationService | 数据源、采集任务、导入批次、差异识别 |
| SecurityApplicationService | 用户、角色、权限、审计、数据范围 |

## 6. 领域服务

| 领域服务 | 适用场景 |
| --- | --- |
| AssetVersioningService | 资产发布、版本快照、版本对比 |
| TermConflictDetectionService | 术语同名、同义、缩写冲突识别 |
| StandardComplianceService | 标准引用和命名合规检查 |
| LineageTraversalService | 血缘路径遍历和上下游分析 |
| ImpactRiskService | 影响等级判定和阻断风险识别 |
| RuleEvaluationService | 治理规则执行和命中结果生成 |
| GovernanceRoutingService | 审批路由和会签建议 |
| ResponsibilityAssignmentService | Owner、Steward、维护人责任分派 |

## 7. 领域事件

关键领域事件：

| 事件 | 触发来源 | 消费方 |
| --- | --- | --- |
| AssetPublished | 资产发布 | catalog、automation、notification |
| AssetChanged | 资产变更 | lineage、impact、notification |
| BusinessTermPublished | 术语发布 | dictionary、modeling、automation |
| BusinessTermDeprecated | 术语废止 | automation、governance、notification |
| StandardPublished | 标准发布 | dictionary、automation、notification |
| LineageConfirmed | 血缘确认 | impact、governance |
| RuleViolationDetected | 规则命中问题 | task、notification |
| GovernanceTaskCreated | 任务生成 | notification |
| ImportBatchCompleted | 导入完成 | automation、notification |
| CollectTaskCompleted | 采集完成 | diff、automation |

首期可使用本地事件总线，后续通过 `DomainEventPublisher` 适配 RocketMQ 或 Kafka。

## 8. 自动化应用协作

自动化模块不直接替代业务模块决策，而是生成建议、任务、报告和通知。

典型流程：

```text
collector 完成采集
-> metadata 保存采集快照
-> automation 执行差异和规则
-> governance 生成确认/整改/审批任务
-> notification 通知责任人
-> metadata 发布确认后的资产版本
```

## 9. 首期边界

首期必须实现：

- metadata、catalog、dictionary、glossary、standard、lineage、governance、automation、collector、security 的核心能力。
- 模块化包结构和依赖规则。
- 领域事件端口。
- 规则、差异、任务、通知的最小闭环。

首期不强制实现：

- 微服务拆分。
- 独立工作流引擎运行时。
- 独立图数据库。
- RocketMQ/Kafka 作为强依赖。
