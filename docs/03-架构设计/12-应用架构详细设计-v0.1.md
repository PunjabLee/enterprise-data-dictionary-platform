# 应用架构详细设计 v0.1

## 1. 设计目标

本文档把应用架构规范落到模块、服务、端口、事件、API 和验收门禁，作为后续开发 Agent 拆分后端、前端和接口工作的实施依据。

## 2. 后端模块设计

| 模块 | 主要应用服务 | 关键端口 | 关键事件 |
| --- | --- | --- | --- |
| system/security | UserApplicationService、RoleApplicationService、AuditApplicationService | PermissionChecker、DataScopeProvider、AuditLogger | PermissionChanged、UserDisabled |
| metadata | MetaModelApplicationService、AssetApplicationService、AssetVersionApplicationService | AssetRepository、MetaModelRepository、AssetQueryPort | AssetImported、AssetPublished、AssetDeprecated |
| catalog | CatalogApplicationService、SubscriptionApplicationService | AssetReadPort、SearchPort、SubscriptionRepository | AssetSubscribed |
| dictionary | DictionaryApplicationService | AssetReadPort、StandardReferencePort、TermReferencePort | DictionaryUpdated |
| modeling | ModelApplicationService、ModelChangeApplicationService | AssetWritePort、ImpactAnalysisPort | ModelChangeSubmitted、ModelPublished |
| glossary | BusinessTermApplicationService、TermConflictApplicationService | TermRepository、TermReferencePort | BusinessTermPublished |
| standard | DataStandardApplicationService、StandardReferenceApplicationService | StandardRepository、StandardReferencePort | DataStandardPublished |
| lineage | LineageApplicationService、ImpactAnalysisApplicationService | LineageRepository、AssetReadPort | LineageConfirmed、ImpactAnalysisGenerated |
| governance | ReviewApplicationService、ApprovalTaskApplicationService、ResponsibilityApplicationService | WorkflowPort、TaskPort、AuditLogger | ReviewApproved、ApprovalRejected、ResponsibilityClaimed |
| automation | RuleApplicationService、DiffApplicationService、TaskApplicationService、NotificationApplicationService | RuleExecutor、TaskRepository、NotificationSender | RuleHit、DiffDetected、TaskCreated |
| collector | DataSourceApplicationService、CollectTaskApplicationService、ImportApplicationService | CollectorAdapter、AssetImportPort | CollectCompleted、ImportBatchCompleted |
| integration | OpenApiApplicationService、WebhookApplicationService、ExternalMappingApplicationService | ApiKeyVerifier、WebhookPublisher、EventAdapter | IntegrationEventPublished |

## 3. 前端模块设计

| 前端区域 | 页面 | 后端上下文 |
| --- | --- | --- |
| 工作台 | 待办、通知、治理概览 | automation、governance |
| 资产目录 | 资产列表、资产详情、订阅关注 | catalog、metadata、security |
| 数据字典 | 表字典、字段字典、定义维护 | dictionary、metadata、glossary、standard |
| 模型管理 | 概念模型、逻辑模型、物理模型、模型变更 | modeling、metadata、governance |
| 标准术语 | 术语库、标准库、发布审批、引用统计 | glossary、standard、governance |
| 血缘分析 | 血缘图、路径分析、影响报告 | lineage、metadata |
| 架构治理 | 评审单、审批任务、检查项、例外豁免 | governance、automation |
| 采集导入 | 数据源、采集任务、导入批次、错误明细 | collector、automation |
| 系统管理 | 用户、角色、权限、数据范围、审计日志 | system/security |

前端约束：

- 页面服务调用必须按后端 owner context 分组。
- 前端权限守卫只做显示与路由控制，后端仍必须强制校验权限。
- 敏感字段、导出按钮、审批操作必须根据权限和数据范围控制显示。

## 4. 关键用例编排

### 4.1 资产导入

```text
ImportController
-> ImportApplicationService
-> TemplateValidator / ImportParser
-> AssetImportPort(metadata)
-> DiffApplicationService(automation)
-> TaskApplicationService(automation)
-> AuditLogger(system/security)
```

约束：

- `collector` 负责批次、模板、解析和外部采集事实。
- `metadata` 负责资产权威写入、版本和关系。
- `automation` 负责差异、规则命中和任务。
- 导入不得直接覆盖已发布业务定义。

### 4.2 架构评审

```text
ReviewController
-> ReviewApplicationService(governance)
-> RuleApplicationService(automation)
-> AssetReadPort(metadata)
-> ImpactAnalysisPort(lineage)
-> ApprovalTaskApplicationService(governance)
-> AuditLogger(system/security)
```

约束：

- `governance` 拥有评审单和审批任务。
- `automation` 生成检查项和规则结果。
- `lineage` 提供影响分析，不拥有评审结论。

### 4.3 模型变更发布

```text
ModelChangeApplicationService(modeling)
-> ImpactAnalysisApplicationService(lineage)
-> ApprovalTaskApplicationService(governance)
-> AssetVersionApplicationService(metadata)
-> DomainEventPublisher
```

约束：

- 变更通过审批后才能发布新版本。
- 发布事件由权威上下文产生。
- 旧版本不得删除，进入历史状态。

### 4.4 血缘确认

```text
LineageApplicationService
-> PermissionChecker
-> LineageRepository
-> ImpactAnalysisCacheInvalidator
-> DomainEventPublisher(LineageConfirmed)
-> AuditLogger
```

约束：

- 关键字段级血缘必须人工确认。
- 正式影响分析默认只使用已确认血缘。

## 5. API 归属示例

| API | Owner Context | 权限 | 审计 |
| --- | --- | --- | --- |
| `GET /api/catalog/assets` | catalog | asset:read + data scope | 否，必要时记录查询统计 |
| `POST /api/collector/import-batches` | collector | import:create | 是 |
| `POST /api/metadata/assets/{id}/publish` | metadata | asset:publish | 是 |
| `POST /api/governance/reviews` | governance | review:create | 是 |
| `POST /api/lineage/relations/{id}/confirm` | lineage | lineage:confirm | 是 |
| `POST /api/standard/standards/{id}/publish` | standard | standard:publish | 是 |
| `GET /api/system/audit-logs` | system/security | audit:read | 否 |

## 6. 事件目录

| 事件 | Producer | Consumer | 幂等键 |
| --- | --- | --- | --- |
| AssetImported | metadata | automation、catalog | eventId 或 batchId + assetId |
| AssetPublished | metadata | catalog、lineage、integration | assetId + version |
| BusinessTermPublished | glossary | dictionary、catalog、integration | termId + version |
| DataStandardPublished | standard | dictionary、catalog、integration | standardId + version |
| LineageConfirmed | lineage | governance、automation、integration | lineageId + version |
| ReviewApproved | governance | metadata、modeling、integration | reviewId + decisionVersion |
| DiffDetected | automation | governance、notification | diffId |
| TaskCreated | automation | notification、integration | taskId |

事件处理规则：

- 事件消费者必须幂等。
- 事件失败必须可追踪，首期可用本地事件表或任务失败表。
- 对外事件由 integration 做契约转换，不直接暴露内部事件结构。

## 7. 数据访问边界

| 数据 | 写入 owner | 读取方式 |
| --- | --- | --- |
| 资产、关系、版本 | metadata | AssetReadPort、Catalog Query |
| 术语 | glossary | TermReadPort |
| 标准 | standard | StandardReadPort |
| 血缘 | lineage | LineageReadPort、ImpactAnalysisPort |
| 评审与审批 | governance | GovernanceQueryPort |
| 任务与通知 | automation | TaskQueryPort、NotificationQueryPort |
| 用户权限审计 | system/security | SecurityPort、AuditQueryPort |

## 8. 架构测试建议

后续工程骨架应提供以下自动化检查：

- domain 不依赖 infrastructure。
- 业务模块不跨上下文依赖 Mapper。
- API controller 只依赖 application service。
- Repository 接口定义在 domain 或 application port，实现放 infrastructure。
- 事件 payload 必须包含 version、tenantId、eventId。
- 敏感导出、审批、发布操作必须调用审计端口。
